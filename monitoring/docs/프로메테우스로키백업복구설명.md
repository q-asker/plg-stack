# Prometheus & Loki 백업·복구 체계

spec `001-prometheus-loki-backup-recovery` 시스템의 스크립트·운영 흐름 진실의 원천.

## 배경 지식

이 문서를 이해하는 데 필요한 핵심 개념·기술 요약. 흐름도만 봐도 각 단계에서 왜 그렇게 설계했는지 이해할 수 있도록.

### Prometheus admin snapshot API

`POST /api/v1/admin/tsdb/snapshot`으로 **TSDB의 하드링크 스냅샷**을 즉시 생성 (spec FR-001 실체).

- 응답: `{"status":"success","data":{"name":"<snap_id>"}}`
- 저장 위치: `<storage.tsdb.path>/snapshots/<snap_id>` (호스트 관점 `/mnt/monitoring/prometheus/snapshots/<snap_id>`)
- **하드링크의 위력**: 실제 데이터 복사가 아니라 inode 참조. 초 단위 완료, 디스크 공간 거의 X (변경 발생 전까지)
- 활성화 조건: `prometheus.command`에 `--web.enable-admin-api` 필수 (T1)
- **위험**: admin 플래그가 `delete_series`, `clean_tombstones` 등 **파괴 API도 함께 노출**시킨다. OCI NSG에서 9090 외부 차단 필수 (RUNBOOK §NSG 점검)

### Loki hardlink `cp -al` 정지 백업

Loki에는 **snapshot API가 없어서** 파일 시스템 직접 백업이 표준 (커뮤니티 관행).

- `POST /flush`로 인메모리 청크를 disk로 밀어내고
- `docker compose stop loki` → `cp -al` hardlink → `docker compose start loki` 흐름
- **`cp -al`의 위력**: 파일 데이터 복사 아니라 inode 참조 하나 더 만들기 → 초 단위 완료
- 정지 시간 실측 2초 (SC-005 60초 한계에 30× 여유)
- Alloy 재시도 큐로 이 시간 로그 손실 없이 흡수

**대안과 비교**:
| 방식 | 문제 |
|---|---|
| 무정지 rsync | compactor가 chunks를 병합·삭제 중이면 정합성 손상 위험 |
| VM 스냅샷 (블록볼륨) | Prometheus까지 통째 백업 = 부분 복원(spec Q1) 불가 |
| Loki를 S3 백엔드로 전환 | 아키텍처 대변경, 요청 카운트 폭증 위험 |
| **`stop → cp -al → start`** (선택) | **몇 초 정지로 정합성 완전 보장, 표준 관행** |

### SHA-256 무결성 검증 (인라인)

`sha256sum`으로 계산한 64자 hex 해시값을 별도 `.sha256` 파일로 저장.

- **인라인 검증 (backup.sh)**: 업로드 직후 READER 프로필로 즉시 GET → 로컬 재계산 → 비교. 실패 시 quarantine 이동 + Slack ERROR
- OCI Object Storage 내부 MD5와 별개로 애플리케이션 계층에서 검증 (FR-009)
- 복원 시(restore.sh) 다운로드 직후에도 `verify_local_hash`로 동일 검증 수행

### Quarantine 격리 패턴

무결성 실패 객체를 **삭제하지 않고 `quarantine/` 접두사로 rename**하여 조사 여지를 남긴다.

- `oci os object rename`은 데이터 이동 없이 메타데이터의 이름만 변경 → 즉시 완료, 추가 저장 X
- 접두사 규칙:
  - `prometheus/`, `loki/` → 정상 경로 (retention 대상)
  - `quarantine/prometheus/*` → 격리 (retention 자동 제외)
- 두 안전장치가 이 접두사에 의존:
  1. lifecycle rule은 `quarantine/` 제외
  2. `retention_cleanup`은 `quarantine/*` prefix 스킵 (backup-common.sh 구현됨)
- 운영자가 원인 분석 후 수동 처리: 확정 삭제 or 원위치 복원

### OCI Object Storage lifecycle policy

버킷의 객체가 특정 조건(예: age)에 도달하면 자동으로 액션(DELETE, ARCHIVE)을 수행하는 규칙.

- T2에서 `delete-after-7d` rule 설정 → 매일 백업된 객체가 7일 후 자동 삭제
- API 요청 카운트 소비 없음 (OCI가 내부에서 처리)
- `retention_cleanup` (backup.sh 마지막 단계)과 **이중 안전망** — lifecycle 실패 시에도 스크립트가 보완
- `delete-after-7d`의 exclusion-patterns에 `quarantine/*`(격리)와 `monthly-archive/*`(영구 보존) 포함
- Standard→Archive 계층화는 미사용 — 무료 20GB가 Standard+Archive 합산이라 비용 이득이 없고 복원 지연만 커진다

### textfile collector (배치 메트릭 노출)

시스템에 상주하지 않는 짧은 스크립트가 Prometheus 메트릭을 노출하는 표준 방법 (FR-010).

- 스크립트가 `<TEXTFILE_DIR>/<name>.prom` 파일에 메트릭을 원자적으로 write하고 종료
- Alloy의 `prometheus.exporter.unix "system"` 블록의 textfile collector가 매 scrape 시 read
- `q_asker_backup.prom` — backup.sh 산출 (last_success_ts, duration, size, loki_downtime, storage_usage/ratio/limit)
- MySQL 백업(`oci_mysql_backup.prom`)과 같은 디렉토리 공존, 파일명·메트릭 접두어 다름

### atomic write (`mktemp + mv -f`)

textfile collector가 반쪽 파일을 읽지 않도록 파일 write를 원자적으로 처리.

```bash
tmp="$(mktemp "${target}.XXXXXX")"      # target.abc123 (.prom 아님, collector 무시)
printf '%s\n' "$content" > "$tmp"
chmod 644 "$tmp"
mv -f "$tmp" "$target"                  # 원자적 rename (rename(2) syscall)
```

- 임시 파일은 확장자 다르므로 collector가 무시 → 반쪽 상태 노출 X
- `mv -f`는 리눅스 커널의 `rename()` 시스템콜, 원자적 실행 보장

### docker compose `--force-recreate`

컨테이너의 volumes·environment·command 등 config는 **생성 시점에 mount namespace로 확정**되어 이후 불변. `restart`는 PID 1만 새로 fork하므로 새 config 미반영.

- `docker compose up -d`도 config 해시 비교로 판단 → 미묘한 변경(예: command 배열)을 놓칠 수 있음
- `--force-recreate`는 판정 우회, 컨테이너를 완전 재생성
- T1에서 `--web.enable-admin-api` 반영, T5에서 alloy volume 반영에 사용

### 호스트 cron (`/etc/cron.d/`) + TZ

시간대 안정성이 중요한 배치 작업의 표준 리눅스 패턴 (spec Q4 KST 03:00).

- `/etc/cron.d/q-asker-backup` 파일이 crond에 자동 등록됨
- 파일 첫 부분에 `CRON_TZ=Asia/Seoul` 명시 → 스케줄을 KST로 해석(시스템 UTC 의존 회피). `TZ`만으로는 스케줄 시각대가 안 바뀐다(잡 환경변수일 뿐)
- 매일 KST 03:00 backup.sh (인라인 검증 + 저장소 임계 알림 포함)
- 로그는 `/var/log/q-asker-backup.log`로 리다이렉트 (logrotate weekly × 4 rotations)

### 매월 아카이브의 시간 창 (옛날 로그 장기 보존)

**슬라이딩 윈도우 한계**: Prometheus/Loki는 자체 retention이 180일이라, 매일 full backup을 유지해도 그 시점의 원본에 없는 데이터는 백업에도 담기지 않는다.

```
[2026-07-04 백업]  2026-01-04 ~ 2026-07-04 (최근 180일)
[2027-01-04 백업]  2026-07-04 ~ 2027-01-04  ← 2026-01 데이터 이미 없음
```

**매월 아카이브 원리**: 매월 1일에 전월 마지막 백업을 별도 접두사로 영구 보관하여, **그 시점의 시간 스냅샷을 무기한 유지**한다.

```
[2026-06 아카이브]  monthly-archive/202606-*.tar.gz
   └─ 2025-12 말 ~ 2026-06 말 (약 180일)

[2026-07 아카이브]  monthly-archive/202607-*.tar.gz
   └─ 2026-01 말 ~ 2026-07 말

[2026-08 아카이브]  monthly-archive/202608-*.tar.gz
   └─ 2026-02 말 ~ 2026-08 말
```

핵심: **오래된 시점의 아카이브를 삭제하지 않으면 그 시점의 데이터가 계속 살아있다**. 2025-12 데이터를 5년 후에도 조회하려면 `202606-*` 파일만 있으면 됨 (그 안에 담김).

Terraform lifecycle에서 `monthly-archive/*` 접두사는 자동 삭제 예외. backup.sh의 저장소 임계 알림(80/90% 2단계)으로 저장소 압박 감지 시 운영자가 RUNBOOK §8-B.4 절차로 오래된 것 수동 정리.

### restore.sh의 rename 롤백 안전망

복원 실패 시 원본을 되돌리기 위해 **rename 기반 격리 후 실패 시 원위치 mv back** (spec Q4).

- 흐름: `mv /mnt/monitoring/<store>` → `<store>.bak.<unix_ts>` → `mkdir <store>` → `tar -xzf` → chown → `docker compose start` → 헬스 폴링
- 실패 발생: 새 dir 삭제 → `.bak.<ts>` 원위치 mv → 재기동
- 성공 시 `.bak.<ts>` 자동 삭제 X (사고 방지, 운영자 수동 정리)

## 스크립트 구성

### backup.sh (T3)

정기 자동 백업 메인 스크립트. 호스트 cron이 매일 KST 03:00 실행한다.

- 인자: `--target=prometheus|loki|both` (기본 both), `--retention-days=N`(보관일 override), `--dry-run`, `--debug`
- Prometheus: snapshot API → tar+gzip → SHA-256 → OCI PUT (WRITER 프로필) → 인라인 verify → 로컬 정리
- Loki: flush → docker compose stop → `cp -al` hardlink → start → tar+gzip → 동일 인라인 검증 흐름
- 각 스토어 실패해도 다른 스토어 계속 진행 (spec Q1: 스토어 독립)
- retention_cleanup 자동 실행 (7일 초과 객체 삭제, quarantine/ 제외)
- check_storage_threshold: 저장소 2단계 경고 — 80% 조기(WARN)·90% 임박(ERROR). 단계 상향 시에만 발송(재발송 억제), 회복 시 상태 갱신. 대응(백업 주기 늘리기)은 cron 조정으로 개발자가 판단 — FR-013
- 결과 → `q_asker_backup.prom` 원자적 갱신 (백업 지표 + storage_usage/ratio/limit)
- 실패 시 Slack ERROR 알림, exit 1

### restore.sh (T4)

특정 시점 백업으로부터 데이터 복원. 재해 복구의 실체 (FR-006, FR-014).

- 인자: `--target=prometheus|loki|both` + `--snapshot=YYYYMMDD-HHMM` (둘 다 필수), `--dry-run`
- `--snapshot` 미지정 시 사용 가능 목록을 자동 조회해 안내 후 exit 2 (Q1=A)
- 흐름: 다운로드 → 로컬 sha256 재검증 → docker compose stop → rename `.bak.<ts>` → tar -xzf `--strip-components=1` → chown(65534/10001) → start → 헬스 폴링 30×10s
- 스토어별 독립 시도 (Q3=B): Prom 실패해도 Loki 계속
- 실패 시 자동 롤백: 새 dir 삭제 → `.bak.<ts>` 원위치 mv → 재기동
- 성공 시 `.bak.<ts>` 보존 (자동 삭제 절대 X, RUNBOOK §.bak 정리 참고)

### lib/backup-common.sh

backup.sh·restore.sh의 공통 함수 라이브러리.

- **로깅/검증**: `log`, `require_env`, `require_cmd`
- **해시**: `hash_file`, `verify_local_hash`
- **OCI 래퍼**: `upload_object`, `download_object`, `delete_object`, `rename_object`, `list_object_keys`, `list_available_snapshots`, `get_bucket_usage_bytes`
- **재시도**: `run_with_retry` (지수 백오프, 기본 3회)
- **인라인 검증**: `verify_object` (backup.sh 업로드 직후 GET+재해시)
- **알림**: `notify_slack` (webhook 부재 시 로그만 남기고 스킵)
- **메트릭**: `write_metrics_atomic` (파일명 파라미터화, mv -f 원자 write)
- **retention**: `retention_cleanup` (7일 초과 삭제, quarantine 제외)
- **헬스 폴링**: `health_poll` (restore.sh용)
- **환경 로드**: `load_env` (정규식 파싱, `.env` 값의 특수문자 안전)

### archive-monthly.sh (매월)

옛날 로그 장기 보존용 아카이브. 호스트 cron이 매월 1일 KST 05:00 실행한다 (backup 03:00 이후).

**아카이브되는 4개 객체 (전월 마지막 백업 기준)**:
```
prometheus/YYYYMMDD-HHMM-prometheus.tar.gz  →  monthly-archive/YYYYMM-prometheus.tar.gz
prometheus/YYYYMMDD-HHMM-prometheus.sha256  →  monthly-archive/YYYYMM-prometheus.sha256
loki/YYYYMMDD-HHMM-loki.tar.gz              →  monthly-archive/YYYYMM-loki.tar.gz
loki/YYYYMMDD-HHMM-loki.sha256              →  monthly-archive/YYYYMM-loki.sha256
```
- 각 tar.gz는 그 시점의 full backup이라 최대 180일치 담김
- 4개 객체 합계 약 1 GiB/월
- 파일명은 원본 timestamp 대신 `YYYYMM` 태그로 단순화 (같은 달의 어느 일이든 하나만 유지)
- 자동 삭제 없음 (Terraform lifecycle에서 `monthly-archive/*` 명시 제외)
- backup.sh의 저장소 임계 알림(80/90% 2단계) → RUNBOOK §8-B.4 절차로 운영자 수동 정리

**동작 흐름**:
1. 전월 태그 계산 (예: 2026-08-01 실행 → `YYYY_MM=202607`)
2. `list_available_snapshots`로 전월(`202607-*`) 최신 timestamp 조회
3. READER로 원본 다운로드 → `verify_local_hash` 무결성 확인 → WRITER로 `monthly-archive/`에 업로드
   (WRITER는 서버사이드 `os object copy` 권한이 없어 GET→PUT 2단계로 수행)
4. 성공/실패 Slack 알림

Standard tier를 유지한다(무료 티어는 20GB가 Standard+Archive 합산이라 Archive 전환의 비용 이득이 없고 복원 지연만 커진다). lifecycle의 `monthly-archive/*` 제외로 영구 보존된다.

### monitoring/cron/q-asker-backup

호스트 cron 참조본. 배포 방법은 파일 상단 코멘트 참조.

- `CRON_TZ=Asia/Seoul` 명시 → 스케줄 시각대 KST 확정 (시스템 timezone 의존 회피)
- 매일 KST 03:00 backup.sh --target=both (인라인 검증 + 저장소 임계 알림 포함)
- 매월 1일 KST 05:00 archive-monthly.sh
- 로그: `/var/log/q-asker-backup.log`로 리다이렉트

### monitoring/logrotate/q-asker-backup

`/var/log/q-asker-backup.log` 회전 규칙.

- weekly × 4 rotations = 최대 4주치 보존
- gzip 압축 + delaycompress (최근 1개는 압축 지연)
- 새 로그 파일 자동 생성 (`create 644 root root`)
- 회전본 접미어 `-YYYYMMDD`

### monitoring/alloy/config.alloy

Alloy 통합 에이전트 설정. T5에서 `prometheus.exporter.unix "system"` 블록에 textfile collector 활성화.

```
enable_collectors = ["textfile"]
textfile {
  directory = "/var/lib/node_exporter/textfile_collector"
}
```

호스트 `/var/lib/node_exporter/textfile_collector`를 컨테이너 안에 ro 마운트해 backup.sh가 write한 `.prom` 파일을 read.

## GameDay (복구 리허설) flow

복구 가능성 보증은 **주기적 GameDay 리허설**로 확보한다.
`gameday.sh`가 실제 `restore.sh`를 비파괴로 실행해 RTO를 실측한다 (SC-004, SC-007).

1. flock으로 backup/restore와 직렬화 (동시 실행 차단)
2. START_TS 기록 → `restore.sh --target=<X> --snapshot=<Y>` 실행
   - 비파괴: rename 흐름으로 원본을 `.bak.<ts>`로 격리 후 복원, 성공해도 원본 보존
3. 헬스 폴링 통과 확인 → END_TS 기록
4. RTO = END_TS - START_TS 실측 (SC-001 target ≤ 14400s / 4h)
   - 실측 기준: Prometheus 179s · Loki 36s (목표 대비 약 80× 여유)
5. 결과를 RUNBOOK 부록 GameDay Log에 기록 (SC-007)

> 백업 무결성 자체는 backup.sh의 **업로드 직후 인라인 검증**(READER GET + 재해시)으로
> 매 백업마다 1회 확인하고, 실패분은 quarantine 격리 + Slack ERROR로 처리한다.
> 저장소 임계 알림(80/90% 2단계)도 backup.sh가 함께 수행한다.
