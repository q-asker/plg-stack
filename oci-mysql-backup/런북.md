# MySQL L2 백업 — 배포·운영 런북

OCI-3 호스트의 `oci-mysql-backup` 서브시스템 배포·갱신·검증·장애 대응 절차.
(백업 로직 배경·복구는 [RESTORE.md](RESTORE.md), 전체 아키텍처는 [../CLAUDE.md](../CLAUDE.md) 참고)

---

## 0. 핵심 구조 — 왜 `git pull`만으론 안 되나

MySQL 백업은 **systemd가 `/opt/oci-mysql-backup/`의 사본을 실행**한다.
`git pull`은 워킹카피(`~/plg-stack/oci-mysql-backup`)만 갱신하므로, **코드 변경은 `/opt`로 재배치해야** 반영된다.

| | 실행 경로 | 코드 반영 |
|---|---|---|
| **PLG 백업** | cron이 `~/plg-stack/...` 직접 실행 | `git pull`만으로 즉시 |
| **MySQL 백업** | systemd가 **`/opt/oci-mysql-backup/` 사본** 실행 | `git pull` + **`update.sh`** 필요 |

배포 파일: `backup.sh`·`restore.sh`·`healthcheck.sh` + `lib/{metrics,metadata,notify}.sh`.
`backup.sh`는 실행 시 `lib/*`를 source하므로 **한 세트로 함께 배포**해야 한다.

관련 위치:
- 코드 배포본: `/opt/oci-mysql-backup/`
- 상태·자격증명: `/var/lib/oci-mysql-backup/` (`state.json`, `.oci/`)
- 환경변수: `/etc/oci-mysql-backup/env` (systemd EnvironmentFile)
- 메트릭: `/var/lib/node_exporter/textfile_collector/oci_mysql_backup.prom`
- systemd: `oci-mysql-backup.service`(oneshot) + `.timer`(6시간 주기)
- 실행 계정: `oci-mysql-backup` (nologin 시스템 사용자)

## 1. 최초 설치 (신규 호스트 1회)

전체 설치(사용자·디렉터리·systemd·env 템플릿 생성)는 `deploy.sh`.

```bash
cd ~/plg-stack/oci-mysql-backup
sudo ./deploy.sh
```

이후 수동 마무리:
```bash
sudo vi /etc/oci-mysql-backup/env                 # MYSQL_*, BUCKET 등 값 채우기
sudo mkdir -p /var/lib/oci-mysql-backup/.oci
sudo cp ~/.oci/config /var/lib/oci-mysql-backup/.oci/config
sudo cp ~/.oci/backup_writer.pem /var/lib/oci-mysql-backup/.oci/
sudo cp ~/.oci/backup_usage_reader.pem /var/lib/oci-mysql-backup/.oci/   # 총량 조회용
sudo chown -R oci-mysql-backup:oci-mysql-backup /var/lib/oci-mysql-backup/.oci
sudo chmod 600 /var/lib/oci-mysql-backup/.oci/*.pem /var/lib/oci-mysql-backup/.oci/config
sudo systemctl enable --now oci-mysql-backup.timer
```

> ⚠️ `.oci/config`의 모든 `key_file` 경로는 **`/var/lib/oci-mysql-backup/.oci/`** 기준이어야 한다.
> `/home/ubuntu/...` 경로가 남아 있으면 `oci-mysql-backup` 계정이 못 읽어 `PermissionError`로 크래시한다(§5.3).

## 2. 코드 업데이트 (일상 — 코드 바꿀 때마다)

```bash
cd ~/plg-stack && git pull origin main
sudo ./oci-mysql-backup/update.sh
```

`update.sh`가 하는 일:
1. 배포 전 소스 전체 `bash -n` (깨진 코드 배포 차단)
2. `/opt`로 코드 동기화 (`← 갱신` / `(동일)` 표시)
3. systemd unit은 **실제 변경분만** 재설치 + `daemon-reload`
4. 배포본 사후 검증 (문법 + lib 존재)

env·자격증명·state는 **건드리지 않는다**(코드만). 실패 시 0이 아닌 종료코드로 중단.

## 3. 검증

**즉시(수동 1회 실행)**:
```bash
sudo systemctl start oci-mysql-backup.service
sudo journalctl -u oci-mysql-backup.service -n 25 --no-pager
```
기대 로그: `[step 1/6] mysqldump...` → `[step 6/6] 무결성 OK...` → `[OK] ... uploaded` → `[storage] 사용량 ...`.
Slack에 `✅ [oci-mysql-backup] SUCCESS` + `저장소 총량 XX%` 도착.

**타이머 상태**:
```bash
systemctl list-timers | grep oci-mysql-backup     # 다음 실행 시각(6h 주기)
```

**최근 백업 객체**:
```bash
oci --profile BACKUP_READER os object list -bn qasker-mysql-backup --limit 5
```

## 4. 롤백 (배포가 잘못됐을 때)

`/opt`는 git이 아니므로, 이전 커밋으로 되돌린 뒤 다시 `update.sh`:
```bash
cd ~/plg-stack
git log --oneline -5 -- oci-mysql-backup/        # 되돌릴 커밋 확인
git checkout <직전-정상-커밋> -- oci-mysql-backup/
sudo ./oci-mysql-backup/update.sh
git checkout HEAD -- oci-mysql-backup/            # 워킹카피 원복(선택)
```
타이머는 다음 주기부터 롤백본으로 실행된다. 진행 중 백업은 flock으로 직렬화되어 안전.

## 5. 장애 대응

### 5.1 Slack 메시지에 변경이 반영 안 됨 (예: 총량%·MB 표기 안 나옴)
→ `/opt` 재배치 누락. `git pull` 후 `sudo ./oci-mysql-backup/update.sh` 실행.
바뀐 파일이 `lib/`까지면 update.sh가 한 번에 처리한다(개별 install 불필요).

### 5.2 마크다운이 `*별표*` 그대로 보임
→ `/opt/.../lib/notify.sh`가 옛 버전(`mrkdwn_in` 없음). `update.sh`로 lib까지 갱신.

### 5.3 `PermissionError: .../home/ubuntu/.oci/*.pem`
→ `/var/lib/oci-mysql-backup/.oci/config`의 `key_file`이 `/home/ubuntu` 경로.
`oci-mysql-backup` 계정은 그걸 못 읽는다. 해당 키를 `/var/lib/oci-mysql-backup/.oci/`로 복사하고 config 경로를 그쪽으로 수정:
```bash
sudo sed -i 's|/home/ubuntu/.oci/|/var/lib/oci-mysql-backup/.oci/|g' /var/lib/oci-mysql-backup/.oci/config
sudo chown -R oci-mysql-backup:oci-mysql-backup /var/lib/oci-mysql-backup/.oci
```

### 5.4 총량 경고/표기가 안 뜸 (`[storage] 총량 조회 실패`)
→ `BACKUP_USAGE_READER` 프로필/키가 서비스 config에 없거나 권한 부족.
`/var/lib/oci-mysql-backup/.oci/config`에 프로필 + 키 존재 확인, `read buckets` 권한 확인:
```bash
sudo -u oci-mysql-backup oci --config-file /var/lib/oci-mysql-backup/.oci/config \
  --profile BACKUP_USAGE_READER os bucket list -c <tenancy-ocid> | jq '.data|length'   # 4 기대
```

### 5.5 백업 실패 Slack (`❌ ERROR stage=dump|upload|...`)
→ `journalctl -u oci-mysql-backup.service -n 50`으로 stage 확인.
`dump`=MySQL 접속/권한, `upload`=OCI 자격증명/네트워크, `checksum/metadata`=디스크·jq.

### 5.6 여러 인스턴스 동시 실행 우려
→ flock(`/var/lock/oci-mysql-backup.lock`)으로 백업/복구/GameDay 직렬화. 락 점유 시 skip(+메트릭). 정상 동작.

## 6. 자주 쓰는 명령

```bash
# 수동 백업 1회
sudo systemctl start oci-mysql-backup.service

# 실시간 로그
sudo journalctl -u oci-mysql-backup.service -f

# 상태 파일(마지막 성공/실패/skip 카운트)
sudo cat /var/lib/oci-mysql-backup/state.json

# 타이머 일시 중지 / 재개
sudo systemctl disable --now oci-mysql-backup.timer
sudo systemctl enable  --now oci-mysql-backup.timer

# 메트릭
cat /var/lib/node_exporter/textfile_collector/oci_mysql_backup.prom
```
